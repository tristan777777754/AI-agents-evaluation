# Task: Phase 11 evaluation credibility

## Current Phase
Phase 11

## Goal
Raise the credibility of run scores and compare conclusions so the workbench can distinguish between directional changes and statistically defensible improvements, while introducing stronger scoring paths that do not rely on the tested agent grading itself.

## In Scope
- Add an `llm_judge` scorer path that evaluates task outputs through a judge model and persists structured score evidence without overwriting existing score records.
- Add a `rubric_based` scorer path that explicitly consumes `dataset_item.rubric_json` when present.
- Add scorer compatibility validation so the default configuration rejects `judge_provider == agent_provider` for judge-based scoring unless an explicit override contract is introduced later.
- Extend compare responses with `sample_size`, `confidence_interval`, `p_value`, and `is_significant`.
- Update compare UI and summary UI to distinguish directional improvement from statistically significant improvement.
- Add deterministic fixtures for judge scoring and compare statistical calculation.
- Update smoke script with a `phase11` path and write a Phase 11 acceptance report.

## Out of Scope
- Full multi-provider model marketplace.
- Re-scoring all historical runs in place.
- Replacing the existing `rule_based` or `keyword_overlap` scorer paths.
- Human-review workflow redesign.
- Multi-scorer tournament logic.

## Acceptance Checks
- When a run uses `scorer_type = "llm_judge"`, then task score records persist judge-backed correctness evidence and retain the original task result unchanged.
- Given a dataset item with `rubric_json.must_do`, `rubric_json.must_not_do`, or `rubric_json.max_steps`, the rubric-based scorer uses those fields in scoring instead of ignoring them.
- When compare is called on two completed runs built from a fixed fixture pair, then the response includes `sample_size`, `confidence_interval`, `p_value`, and `is_significant` values that can be recomputed from the fixture.
- When compare UI renders a run pair with positive delta but `is_significant = false`, then the UI labels the result as directional or inconclusive rather than as a confirmed improvement.
- `./scripts/smoke.sh phase11` executes a deterministic credibility path and fails clearly if statistical fields or judge score evidence are missing.

## Contract Checks
- Existing scorer types remain valid; `llm_judge` and `rubric_based` are additive scorer paths.
- Compare response only adds credibility-related fields; existing compare keys remain backward-compatible.
- Canonical run, task, score, review, and compare entity names remain unchanged.
- Judge-based scoring must not overwrite or mutate historical score payloads generated by older scorer types.

## Rollback / Recovery Notes
- If judge scoring destabilizes existing score serialization, revert the new scorer path and keep older scorer types as the only active options.
- If statistical summary fields break compare consumers, move them into an additive nested block while preserving the original compare envelope.
- If rubric scoring introduces ambiguous fixture expectations, narrow the rubric fixture before changing canonical score semantics.

## Stop And Ask Conditions
- Judge scoring requires changing the canonical `score` schema in a way that breaks Phase 6 review records or Phase 9 calibration records.
- Statistical significance requires introducing new infrastructure or external dependencies that materially change the project’s established stack without a documented approval.
- The judge compatibility rule conflicts with already committed Phase 7-10 contracts for provider configuration.
